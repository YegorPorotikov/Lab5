Тема: Система контролю версій Git та співпраця через GitHub

Загальний опис теми
Git — це розподілена система контролю версій (VCS, Version Control System), створена Лінусом Торвальдсом у 2005 році для розробки ядра Linux. Сьогодні Git є де-факто стандартом для контролю версій у програмному забезпеченні та використовується мільйонами розробників по всьому світу.
Система контролю версій дозволяє відстежувати зміни в коді з часом, повертатися до попередніх версій, працювати над проектом командою без конфліктів, експериментувати з новими функціями в ізольованих гілках, та багато іншого. Git зберігає повну історію проекту, що дозволяє відповісти на питання "хто, що, коли і чому змінив" у будь-який момент часу.
Розуміння Git є критично важливим для сучасних розробників, незалежно від спеціалізації. Це не просто інструмент для збереження коду — це фундамент для ефективної командної роботи, continuous integration/deployment, code review процесів та професійної розробки ПЗ.

1. Основи Git
Git та GitHub?Git — інструмент (програма). GitHub — сервіс (сайт) для Git.
"Розподілена" (Distributed)?
Кожен розробник має повну копію всієї історії проєкту локально.
git init та git clone?init — створює новий порожній репозиторій. clone — копіює існуючий.
2. Staging Area та Commits
Staging Area?
Це "чернетка" для комміту. Дозволяє вибірково готувати зміни.
git add . та git add -A?add . — нові/змінені файли. add -A — всі (включаючи видалені).
Гарне commit message?
Короткий (до 50 символів), в наказовому способі (наприклад, "Fix bug").
Виправити останній коміт?git commit --amend (якщо ще не відправили push).
3. Гілки та злиття
Гілка (Branch) та чому "легковажні"?
Це просто вказівник на комміт. Легкі, бо не копіюють всі файли.
git merge та git rebase?merge — об'єднує історії (створює "merge-комміт"). rebase — переписує історію, роблячи її лінійною.
Fast-forward merge?
Просте "перемотування" вказівника гілки вперед, без створення нового комміту.
Merge конфлікти?
Коли дві гілки змінили один і той самий рядок в одному файлі.
Чому не можна робити rebase публічних комітів?
Це ламає історію для всіх інших, хто вже завантажив код.
4. Віддалені репозиторії
Що таке origin?
Стандартна назва (псевдонім) для віддаленого репозиторію.
git fetch та git pull?fetch — завантажує зміни (але не зливає). pull — завантажує і зливає (fetch + merge).
git push --force?Примусово перезаписує історію на сервері. Дуже небезпечно, можна стерти чужу роботу.
Видалити віддалену гілку?git push origin --delete <назва-гілки>
